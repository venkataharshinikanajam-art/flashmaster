# 00 — Start Here

Welcome to FLASHMASTER. If you're reading this, you're about to build your first real full-stack web app — and learn full-stack development properly along the way.

This document is your **map**. Read it once now, and come back whenever you feel lost.

---

## Who this is for

**Harsh.** A first-time full-stack developer who knows basic C and Python, and is completely new to JavaScript, Node, React, MongoDB, terminals, git, and web development in general. Don't worry — so is everyone when they start.

## What we're building

**FLASHMASTER — Exam Helper App**, a web application that:
- Lets students upload study materials (PDFs, text, images)
- Auto-generates flashcards from them
- Organizes everything by subject/topic
- Helps plan study schedules around exam dates
- Tracks progress and shows analytics
- Has admin features for managing users and content

It's your university project. The full brief is in `FLASHMASTER – Exam Helper App.docx`.

## How this project teaches you

We learn by doing, but we never code something we haven't **understood first**. Every phase follows the same rhythm:

1. **Concept** — what is this thing? (with a real-life analogy)
2. **Vocabulary** — new words added to the glossary
3. **Example prompts** — how to ask Claude for help in this phase
4. **Build** — the hands-on part, one small step at a time
5. **Checkpoints** — quick questions so you know you understood
6. **Track** — update `PROGRESS.md` and `TASKS.md`

You are not expected to know anything upfront. The project is designed to teach you.

## Your first day (the right order)

1. **Read this file** (you're doing it).
2. **Read `docs/02-how-we-work.md`** — the daily rhythm.
3. **Read `docs/ai-prompting-guide.md`** — how to talk to Claude effectively.
4. **Open `docs/phases/phase-00-foundations.md`** — your first phase playbook.
5. **Follow Phase 0 step-by-step.** It installs your tools and teaches you the basics.

That's it. Do not try to jump ahead to Phase 5 because it "sounds cool". Each phase builds on the previous one — skipping is how you end up confused.

## Getting help

You have three ways to get unstuck:

1. **Ask Claude directly in chat.** Be specific. Paste errors verbatim. Say "teach me before coding" if you want an explanation.
2. **Ask a specialized agent:**
   - **teacher** — for new concepts
   - **debugger** — when something breaks
   - **code-reviewer** — at the end of a phase
   - **researcher** — when choosing between libraries or checking if something is current
3. **Check the docs** in this folder first. If the glossary already defines a word, use it. If `learning/` already explains a concept, re-read it.

## Ground rules

- **No copy-paste without understanding.** If you don't know what a line does, ask Claude to explain it.
- **Run commands yourself after Claude demos them.** Typing builds muscle memory.
- **Break things on purpose sometimes.** See what the error says. Errors are your teachers too.
- **Commit small and often.** One working change = one commit. You'll love past-you for this.
- **Don't skip phases.** Seriously.
- **Ask stupid questions.** There are no stupid questions in a field this vast. The only mistake is pretending to understand.
